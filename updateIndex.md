# Index File Generation Instructions

## Vault root

`VAULT_ROOT` = the directory containing this `updateIndex.md` file (the current Obsidian vault root). All paths below are relative to it. Do not hardcode an absolute path — resolve `VAULT_ROOT` from the location of this file at execution time.

## Purpose

Create or update `VAULT_ROOT/index.md` as a navigation hub for all markdown files in the vault.

## Execution: parallel agents

This task is **embarrassingly parallel** — each top-level subdirectory of the vault is independent. Discover them at runtime, then dispatch one `Explore` agent per subdirectory in a **single message with multiple tool calls** so they run concurrently.

### Step 1 — Discover the structure

At `VAULT_ROOT`:

1. List `*.md` files at depth 1 only (directly in the root). Record their titles/descriptions yourself — these become the "Top-Level Files" section. Exclude `index.md`, `updateIndex.md`, `CLAUDE.md`, and anything starting with `.`.
2. List immediate child directories. Exclude any directory starting with `.` (e.g. `.obsidian`, `.git`), any `node_modules`, and `inbox/` (see Exclude section). Whatever remains is the set of category directories.

Do not assume a fixed list. The vault structure can change — drive everything from this discovery.

### Step 2 — Dispatch one agent per subdirectory (parallel)

Send a single message containing one `Agent` tool call per discovered subdirectory. Use `subagent_type: "Explore"` with `thoroughness: "medium"`.

**Prompt template** (substitute `<CATEGORY>` and `<VAULT_ROOT>`):

> Walk `<VAULT_ROOT>/<CATEGORY>/` recursively and produce an inventory of every `.md` file. For each file, report:
> - Relative path from vault root (e.g. `<CATEGORY>/sub/foo.md`)
> - Title (from YAML frontmatter `title`, else the H1, else the filename)
> - One-line description (from frontmatter `description` if present, else distilled from the 2–3 line context block under the H1, max ~120 chars)
> - Subdirectory grouping (the immediate child folder under `<CATEGORY>`, or `_root` if directly in the category dir)
>
> Skip: any dotfile directory, binary files, and any file with `status/archived` in its tags. Include drafts but mark them `(draft)`.
>
> Return a flat markdown list grouped by subdirectory, alphabetical within each group. No prose summary — just the inventory.

### Step 3 — Assemble `index.md`

Merge agent reports into a single file. The category sections are emitted in the order the subdirectories were discovered (alphabetical is fine). Skip any category that returned zero entries — don't emit an empty header.

Skeleton:

```markdown
---
title: Vault Index
aliases: [index, home]
tags: [type/index, status/living]
date: <today>
status: living
type: index
author: claude
---

# Vault Index

Navigation hub for this vault. Auto-generated — see `updateIndex.md` for regeneration instructions.

## Top-Level Files
<entries from Step 1>

## <category-1>
<entries from agent 1, grouped by ### subdirectory>

## <category-2>
...

## Sources
| Source | Note |
|---|---|
| Vault filesystem scan | Generated by `updateIndex.md` workflow |

## Gaps
- <list any agent failures or empty subdirectories worth flagging>
```

## Formatting rules

- **Links**: Obsidian wikilinks `[[relative/path/note|Display Title]]`. Not markdown links.
- Each entry: `- [[path|Title]] — brief description`
- `##` for category headings, `###` for subdirectory groupings within a category
- Alphabetical within each (sub)section
- Skip empty sections silently

## Exclude

- Any dotfile directory (`.obsidian/`, `.git/`, etc.)
- `node_modules/`
- `inbox/` — staging area for unsorted notes; not part of the curated knowledge base
- Binary files (PDFs, images, archives)
- `index.md`, `updateIndex.md`, `CLAUDE.md`

## Update vs. replace

**Replace**, don't merge. The index reflects current filesystem state — stale entries from prior runs are not preserved. If a previously-indexed file is gone, it's gone.
